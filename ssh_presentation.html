🤔 O que é SSH?

Secure Shell - Protocolo para acesso remoto seguro
Substitui protocolos inseguros como Telnet, rlogin, rsh
Criptografia forte para confidencialidade
Protocolo TCP na porta 22

💡 Analogia Simples:

Telegrama comum (todos podem ler)
⬇️
Carta lacrada e assinada
⬇️
Só o destinatário consegue ler

SSH transforma comandos "telegrama" em "cartas lacradas"!

🔐 Criptografia no SSH

SSH usa combinação de criptografia simétrica e assimétrica
Chaves públicas para autenticação e troca inicial
Chaves simétricas para dados da sessão
Algoritmos modernos e seguros

🔑 Processo de Criptografia SSH

1. Handshake inicial:
Cliente ←→ Servidor: Versões e algoritmos suportados

2. Troca de chaves (Diffie-Hellman):
Cliente + Servidor = Chave compartilhada secreta

3. Autenticação:
Servidor: Prova identidade com chave privada
Cliente: Senha ou chave privada

4. Sessão criptografada:
Todos os dados com AES/ChaCha20
Hash HMAC para integridade

🔑 Criptografia Assimétrica

Para que serve:
• Autenticação inicial
• Troca segura de chaves
• Verificação de identidade
Algoritmos:
• RSA (2048-4096 bits)
• ECDSA (Elliptic Curve)
• Ed25519 (moderno)

🔒 Criptografia Simétrica

Para que serve:
• Dados da sessão
• Comandos e respostas
• Velocidade alta
Algoritmos:
• AES-128/256
• ChaCha20-Poly1305
• 3DES (legado)

🔐 Analogia da Criptografia

Cofre com duas fechaduras:
🔑 Chave pública = Abrir para colocar
🗝️ Chave privada = Abrir para retirar

Só quem tem a chave privada consegue ler!

⚡ Por que Criptografia Híbrida?

Assimétrica: Segura mas lenta
Simétrica: Rápida mas precisa trocar chave
SSH: Usa assimétrica para trocar simétrica = Melhor dos dois!

🤝 Processo de Conexão SSH

Handshake em múltiplas etapas para máxima segurança
Verificação da identidade do servidor
Negociação de algoritmos criptográficos
Estabelecimento de canal seguro
1. Identificação: Cliente conecta na porta 22
2. Versão: Troca de versões SSH suportadas
3. Algoritmos: Negociação de métodos de criptografia
4. Chave do servidor: Servidor apresenta sua chave pública
5. Diffie-Hellman: Geração de chave compartilhada
6. Verificação: Cliente verifica identidade do servidor
7. Autenticação: Cliente prova sua identidade
8. Canal seguro: Comunicação criptografada iniciada

🔍 Exemplo de Handshake

# 1. Conexão inicial
Cliente → Servidor: SSH-2.0-OpenSSH_8.9
Servidor → Cliente: SSH-2.0-OpenSSH_8.9

# 2. Algoritmos suportados
Cliente → Servidor: [AES-256, RSA-2048, HMAC-SHA256]
Servidor → Cliente: [AES-256, RSA-2048, HMAC-SHA256] ✅

# 3. Chave do servidor
Servidor → Cliente: Minha chave pública RSA é...
Cliente: ⚠️ Verificando se confio nesta chave...

# 4. Diffie-Hellman
Cliente + Servidor: Calculando chave compartilhada secreta

# 5. Pronto!
🔒 Canal seguro estabelecido!

⚠️ "Host Key Verification Failed"

Primeira conexão: SSH pergunta se você confia no servidor
Chave mudou: Pode indicar ataque "man-in-the-middle"
Ação: Sempre verificar com administrador antes de aceitar!

SSH oferece múltiplos métodos de autenticação
Cada método com diferentes níveis de segurança
Possível combinar métodos (autenticação multifator)
Configurável no servidor e cliente

Método mais comum
Senha tradicional
Criptografada no SSH
Vulnerável a ataques

🔄 Autenticação por Chave Pública

1. Preparação:
Cliente gera par de chaves (pública/privada)
Chave pública é copiada para o servidor

2. Durante conexão:
Servidor: "Prove que tem a chave privada!"
Cliente: Assina mensagem com chave privada
Servidor: Verifica assinatura com chave pública
Servidor: "OK, você é quem diz ser!" ✅

✅ Vantagens da Chave Pública

• Sem senha na rede
• Impossível interceptar
• Ideal para scripts
• Pode ter passphrase adicional
• Muito mais seguro

🔧 Configuração do SSH

Configuração do servidor em /etc/ssh/sshd_config
Configuração do cliente em ~/.ssh/config
Múltiplas opções de segurança
Logs detalhados para monitoramento

🖥️ Configuração do Servidor

# /etc/ssh/sshd_config

# Porta do SSH
Port 22

# Versão do protocolo
Protocol 2

# Autenticação
PasswordAuthentication yes
PubkeyAuthentication yes
PermitRootLogin no

# Segurança
MaxAuthTries 3
ClientAliveInterval 300
ClientAliveCountMax 2

# Logs
LogLevel INFO
SyslogFacility AUTH

💻 Configuração do Cliente

# ~/.ssh/config

# Servidor de trabalho
Host trabalho
HostName servidor.empresa.com
User meuusuario
Port 2222
IdentityFile ~/.ssh/trabalho_rsa

# Servidor pessoal
Host casa
HostName 192.168.1.100
User admin
IdentityFile ~/.ssh/casa_ed25519

# Configurações globais
Host *
ServerAliveInterval 60
ServerAliveCountMax 3

🔒 Hardening de Segurança

# Configurações recomendadas para produção

# Desabilitar senha para root
PermitRootLogin no

# Apenas autenticação por chave
PasswordAuthentication no
ChallengeResponseAuthentication no

# Limitar usuários
AllowUsers admin backup
DenyUsers root nobody

# Mudar porta padrão
Port 2222

# Desabilitar recursos desnecessários
X11Forwarding no
AllowTcpForwarding no
PermitTunnel no

💻 Comandos SSH Essenciais

SSH oferece muito mais que apenas acesso remoto
Transferência de arquivos com SCP e SFTP

Execução de comandos únicos

🔗 Conexão Básica

# Conexão simples
$ ssh usuario@servidor

# Especificar porta
$ ssh -p 2222 usuario@servidor

# Usar arquivo de chave específico
$ ssh -i ~/.ssh/chave usuario@servidor

# Conexão com comando único
$ ssh usuario@servidor 'ls -la'

# Debug de conexão
$ ssh -v usuario@servidor

📁 Transferência de Arquivos

# SCP - Secure Copy
$ scp arquivo.txt user@server:~/
$ scp user@server:~/arquivo.txt .
$ scp -r pasta/ user@server:~/backup/

# SFTP - Interactive
$ sftp user@server
sftp> ls
sftp> get arquivo.txt
sftp> put local.txt
sftp> exit

# rsync sobre SSH
$ rsync -avz -e ssh pasta/ user@server:~/

🔄 Port Forwarding (Tunelamento)

# Forward local → remoto
# Acessa serviço remoto via localhost
$ ssh -L 8080:localhost:80 user@server
# Agora localhost:8080 → server:80

# Forward remoto → local
# Servidor acessa serviço local
$ ssh -R 9000:localhost:3000 user@server
# Agora server:9000 → localhost:3000

# SOCKS Proxy
$ ssh -D 1080 user@server
# Proxy SOCKS em localhost:1080

🔑 Gerenciamento de Chaves

# Gerar nova chave
$ ssh-keygen -t ed25519

# Copiar chave pública
$ ssh-copy-id user@server

# Listar chaves carregadas
$ ssh-add -l

# Carregar chave no agent
$ ssh-add ~/.ssh/chave

📋 Informações

# Versão SSH
$ ssh -V

# Algoritmos suportados
$ ssh -Q cipher
$ ssh -Q mac
$ ssh -Q kex

# Testar configuração
$ ssh -T git@github.com

🔧 Utilitários

# Verificar chave do servidor
$ ssh-keyscan servidor

# Fingerprint da chave
$ ssh-keygen -lf chave.pub

# Converter formato de chave
$ ssh-keygen -p -f chave

# Remover chave conhecida
$ ssh-keygen -R servidor

🚨 Ameaças Comuns

Brute Force:
Tentativas massivas de senha
Robôs testam senhas comuns
Porta 22 é muito visada
Man-in-the-Middle:
Interceptação da conexão
Certificados falsos
Redes WiFi inseguras
Chaves Comprometidas:
Chaves privadas vazadas
Sem passphrase
Não revogadas

✅ Contramedidas

Contra Brute Force:
• Mudar porta padrão
• Fail2ban
• Senhas fortes
• Rate limiting
Contra MITM:
• Verificar fingerprints
• Certificate pinning
• VPN para redes públicas
• Known_hosts atualizado
Gestão de Chaves:
• Rotação regular
• Passphrase sempre
• Revogação imediata
• Hardware tokens

🔒 Configuração de Segurança Máxima

# /etc/ssh/sshd_config - Configuração hardened

# Protocolo e porta
Protocol 2
Port 2222 # Não usar 22

# Autenticação
PasswordAuthentication no
PubkeyAuthentication yes
AuthenticationMethods publickey
PermitRootLogin no
MaxAuthTries 2

# Criptografia forte
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs hmac-sha2-256-etm@openssh.com,hmac-sha2-512-etm@openssh.com
KexAlgorithms curve25519-sha256@libssh.org,diffie-hellman-group16-sha512

# Limites de tempo
ClientAliveInterval 300
ClientAliveCountMax 2
LoginGraceTime 30

# Controle de acesso
AllowUsers admin backup
DenyUsers root nobody
AllowGroups ssh-users

❌ Problemas Comuns

"Connection refused"
• SSH daemon não rodando
• Porta errada
• Firewall bloqueando
• Rede inacessível
"Permission denied"
• Senha incorreta
• Chave pública incorreta
• Usuário sem permissão
• Autenticação desabilitada
"Host key verification failed"
• Chave do servidor mudou
• Possível ataque MITM
• Known_hosts incorreto
• Primeiro acesso

🔍 Comandos de Diagnóstico

# Debug da conexão
$ ssh -v user@server
$ ssh -vv user@server
$ ssh -vvv user@server

# Testar conectividade
$ telnet server 22
$ nc -zv server 22

# Verificar serviço
$ systemctl status sshd
$ sudo sshd -T # Teste config

# Logs do servidor
$ tail -f /var/log/auth.log
$ journalctl -u sshd -f

# Testar chaves

$ ssh -vvv admin@servidor.empresa.com

7. Logs de erro detalhados (-vvv)

# Script para deploy em múltiplos servidores
#!/bin/bash

SERVERS="web1 web2 web3 db1"
DEPLOY_DIR="/var/www/app"

for server in $SERVERS; do
echo "🚀 Deploy em $server..."

# Upload do código
rsync -avz --delete app/ $server:$DEPLOY_DIR/

# Reiniciar serviços
ssh $server "sudo systemctl restart nginx php-fpm"

# Verificar status
ssh $server "curl -s http://localhost/health" || echo "❌ Falha em $server"
done

echo "✅ Deploy concluído!"

🎯 Cenário: Servidor Web em Produção

Objetivo: Configurar acesso SSH seguro para um servidor web Ubuntu 22.04
Requisitos: Apenas chaves públicas, porta personalizada, logs auditados

🔧 1. Configuração Inicial

# No servidor (como root inicialmente)
$ apt update && apt upgrade -y
$ apt install fail2ban ufw -y

# Criar usuário administrativo
$ adduser admin
$ usermod -aG sudo admin

# Configurar firewall
$ ufw default deny incoming
$ ufw default allow outgoing
$ ufw allow 2222/tcp # SSH customizado
$ ufw allow 80/tcp # HTTP
$ ufw allow 443/tcp # HTTPS
$ ufw enable

🔑 2. Configuração SSH

# /etc/ssh/sshd_config
$ sudo nano /etc/ssh/sshd_config

# Mudanças importantes:
Port 2222
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
MaxAuthTries 3
MaxSessions 10
ClientAliveInterval 300
ClientAliveCountMax 2
LogLevel VERBOSE

# Reiniciar SSH
$ sudo systemctl restart sshd
3. No cliente: Gerar e configurar chaves
# Gerar chave Ed25519 (moderna e segura)
$ ssh-keygen -t ed25519 -C "admin@servidor-web"
Enter file in which to save the key: /home/user/.ssh/servidor_web
Enter passphrase: **************

# Copiar chave pública (ainda com senha, última vez)
$ ssh-copy-id -i ~/.ssh/servidor_web.pub -p 2222 admin@servidor.empresa.com

# Configurar ~/.ssh/config
Host servidor-web
HostName servidor.empresa.com
Port 2222
User admin
IdentityFile ~/.ssh/servidor_web
ServerAliveInterval 60
4. Teste e verificação:
# Teste de conexão
$ ssh servidor-web
✅ Conectado sem senha!

# Verificar configuração
admin@servidor:~$ sudo sshd -T | grep -E "(port|permit|password)"
port 2222
permitrootlogin no
passwordauthentication no

# Monitorar logs
admin@servidor:~$ sudo tail -f /var/log/auth.log
Jun 28 15:30:25 servidor sshd[1234]: Accepted publickey for admin from 203.0.113.10 port 52341 ssh2: ED25519 SHA256:abc123...

✅ Resultado Final

🔒 Acesso apenas por chave pública
🛡️ Porta não-padrão (2222)
📊 Logs detalhados
🚫 Root login desabilitado
🔥 Firewall configurado
