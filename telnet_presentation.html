🤔 O que é TELNET?

TErminaL NETwork - Protocolo para acesso remoto
Aplicação TCP/IP padrão para serviços de terminal virtual
Programa cliente/servidor de finalidade genérica
Permite executar qualquer aplicação em computador remoto

💡 Propósito do TELNET:

Terminal Local se comporta
⬇️
EXATAMENTE
⬇️
como se fosse terminal do sistema remoto

🎯 Cenários de Uso:

Estudante acessa laboratório da universidade de casa
Administrador gerencia servidores remotamente
Desenvolver programas cliente/servidor específico para cada necessidade seria impossível
TELNET oferece solução genérica e flexível

⚠️ Observação Importante

TELNET foi desenvolvido quando a maioria dos sistemas operacionais funcionava em ambiente de tempo compartilhado (time sharing)

⏰ Ambiente de Tempo Compartilhado

Computador grande suporta vários usuários simultaneamente
Interação via terminal (teclado + monitor + mouse)
Cada usuário tem identificação (login) e senha
Recursos compartilhados com controle de acesso

🖥️ Componentes do Sistema

Terminal
Teclado + Monitor
(Interface física)
Driver do Terminal
Aceita digitações
(Software de interface)
Sistema Operacional
Interpreta comandos
(Núcleo do sistema)

🔐 Processo de Autenticação

Identificação do usuário → Verificação de senha → Definição de direitos de acesso
Sistema controla quem pode acessar quais recursos

🖥️ Microcomputador como Terminal

Um microcomputador pode simular terminal usando emulador de terminais
Software especial faz PC se comportar como terminal "burro"

🏠 Login Local vs 🌐 Login Remoto

🏠 Login Local

1. Usuário digita no terminal
2. Driver do terminal aceita caracteres
3. Repassa para sistema operacional
4. SO interpreta e executa programa
Fluxo Simples:
Terminal → Driver → SO → Programa

🌐 Login Remoto

1. Usuário digita no terminal
2. SO local aceita mas NÃO interpreta
3. Cliente TELNET converte para NVT
4. Transmite via Internet
5. Servidor TELNET recebe
6. Pseudodriver converte para SO remoto
7. SO remoto interpreta e executa
Fluxo Complexo:
Terminal → Cliente TELNET → Internet → Servidor TELNET → Pseudodriver → SO Remoto

🔧 Pseudodriver de Terminal

Por que é necessário? SO não foi projetado para interpretar caracteres de servidor TELNET
Solução: Pseudodriver faz conversões adicionais, simulando terminal local

🖥️ Terminal Virtual de Rede (NVT)

Solução para sistemas heterogêneos incompatíveis
Interface universal entre diferentes computadores
Cada sistema tem convenções diferentes para mesma função

🎯 Problema dos Sistemas Heterogêneos:

Sistema DOS
Fim de arquivo = Ctrl+Z
Sistema UNIX
Fim de arquivo = Ctrl+D

Mesmo comando, caracteres diferentes!

🔄 Como o NVT Resolve o Problema

Cliente TELNET
Converte caracteres locais
⬇️
para formato NVT
FORMATO NVT
Interface Universal
📡
Transmitido pela rede
Servidor TELNET
Converte NVT
⬇️
para formato local

💡 Vantagem do NVT

Sem NVT: Seria necessário conhecer cada tipo de computador remoto
Com NVT: Um formato universal funciona com qualquer sistema

📝 Conjunto de Caracteres NVT

NVT implementa dois conjuntos de caracteres de 8 bits
Separação clara entre dados e comandos de controle

📄 Caracteres de Dados

7 bits de ordem inferior = ASCII padrão
Bit de ordem superior = 0
Conteúdo normal das mensagens
Compatível com sistemas legados
Exemplo de dados:
01001000 = 'H' (ASCII 72)
01100101 = 'e' (ASCII 101)
01101100 = 'l' (ASCII 108)
01101100 = 'l' (ASCII 108)
01101111 = 'o' (ASCII 111)

⚙️ Caracteres de Controle

Bit de ordem superior = 1
Comandos especiais do protocolo
Controle da comunicação
Negociação de opções
Exemplos de controle:
11111111 = IAC (255)
11110100 = IP (244)
11110111 = EC (247)
11111000 = EL (248)

🎯 Identificação Automática

Sistema examina bit mais significativo:
0 = Dados (processa como texto normal)
1 = Controle (interpreta como comando)

💡 Vantagem da Separação

Não há confusão entre dados do usuário e comandos do protocolo
Sistema sempre sabe como interpretar cada byte recebido

⌨️ Caracteres de Controle NVT

Caractere Código Binário Significado
IAC 255 11111111 Interpret As Control - Próximo é comando
IP 244 11110100 Interrupt Process - Interrompe processo
AO 245 11110101 Abort Output - Cancela saída pendente
EC 247 11110111 Erase Character - Apaga caractere (backspace)
EL 248 11111000 Erase Line - Apaga linha inteira
AYT 246 11110110 Are You There? - Teste de conectividade

💻 Exemplo Prático de Uso

Usuário digita: cat filea e percebe erro

Pressiona backspace e digita: 1

Sequência enviada:
c a t f i l e a <IAC><EC> 1

O backspace é convertido em IAC + EC (dois caracteres de controle)

🔧 Função do IAC

IAC = "Interpret As Control" - Marca início de sequência de controle
Sem IAC: Sistema não saberia distinguir dados de comandos
Com IAC: Sistema sabe que próximo byte é comando especial

🔀 Embedding de Comandos

TELNET usa apenas uma conexão TCP (porta 23)
Mesma conexão para dados e comandos de controle
Comandos são "embutidos" no fluxo de dados
IAC distingue controle de dados

💡 Exemplo Detalhado de Embedding

Cenário: Usuário quer exibir arquivo "file1" mas digita "filea" por engano

1. Usuário digita: cat filea
2. Percebe erro e pressiona backspace
3. Digita: 1
Sequência de bytes transmitida:
[c] [a] [t] [ ] [f] [i] [l] [e] [a] [IAC] [EC] [1]
↑ ↑ ↑ ↑
dados normais controle dados

Resultado: Servidor interpreta backspace e executa: cat file1

⚡ Vantagens do Embedding

Uma única conexão TCP (mais simples)
Menor overhead de rede
Comandos sincronizados com dados
Não há problema de ordem entre conexões

⚠️ Limitação Importante

Na implementação padrão do TELNET, usuário NÃO pode editar localmente
Toda edição (backspace, etc.) ocorre no servidor remoto

⚙️ Negociação de Opções

TELNET permite negociar recursos extras
Opções para terminais mais sofisticados
Usuários de terminais simples usam recursos-padrão
Negociação antes ou durante o uso do serviço

📋 Comandos de Negociação

WILL (251) Oferece-se para habilitar opção
WONT (252) Rejeita ou oferece-se para desabilitar
DO (253) Aprova oferta ou solicita habilitação
DONT (254) Rejeita oferta ou solicita desabilitação

🔧 Opções Comuns

Echo (1) Eco dos caracteres
Binary (0) Transmissão binária 8 bits
Terminal Type (24) Tipo de terminal
Line Mode (34) Modo de linha

💬 Exemplo de Negociação - Echo

Cliente: IAC DO ECHO
"Por favor, habilite a opção de eco"
Servidor: IAC WILL ECHO
"Ok, vou habilitar a opção de eco"

Resultado: Servidor passa a fazer eco dos caracteres de volta ao cliente

🔄 Processo de Negociação

Oferecer: WILL = "Posso fazer isso?" → Resposta: DO/DONT
Solicitar: DO = "Por favor, faça isso" → Resposta: WILL/WONT

🔧 Negociação de Subopções

Algumas opções requerem parâmetros adicionais
Exemplo: tipo específico de terminal ou velocidade
Subopções permitem passar informações detalhadas

📝 Caracteres de Subopção

SB (250) Início da Subopção
SE (240) Fim da Subopção

🖥️ Formato Geral

IAC SB <opção> <parâmetros> IAC SE

💻 Exemplo: Tipo de Terminal

1. Cliente oferece:
IAC WILL TERMINAL_TYPE
2. Servidor aceita:
IAC DO TERMINAL_TYPE
3. Cliente especifica:
IAC SB TERMINAL_TYPE "VT100" IAC SE

🎯 Exemplo Completo de Subopção

Sequência completa para definir terminal VT100:

IAC WILL TERMINAL_TYPE (Cliente oferece opção)
IAC DO TERMINAL_TYPE (Servidor aceita)
IAC SB TERMINAL_TYPE 'V' 'T' '1' '0' '0' IAC SE ↑ ←── Parâmetros ──→ ↑
Início Fim

⚡ Outros Exemplos de Subopções

Velocidade do terminal: 9600, 19200, 38400 bps
Tamanho da tela: 80x24, 132x43 caracteres
Capacidades especiais: cores, gráficos, som

🔄 Modos de Operação TELNET

📝 Modo Padrão

Usado se nenhum outro for negociado
Eco feito localmente pelo cliente
Linha completa enviada de uma vez
Cliente não transmite até linha estar pronta
👍 Vantagem:
Menor tráfego de rede
👎 Desvantagem:
Sem feedback imediato

⌨️ Modo Caractere

Cada caractere enviado imediatamente
Servidor faz eco de volta
Interação mais responsiva
Alto overhead de rede
👍 Vantagem:
Resposta imediata
👎 Desvantagem:
3 segmentos TCP por caractere!
(requisição + eco + confirmação)

📏 Modo Linha

Proposto para compensar ineficiências
Edição de linha feita localmente
Cliente transmite linha completa
Equilibra eficiência e usabilidade
👍 Vantagem:
Melhor dos dois mundos
🎯 Status:
Modo preferido atualmente

⚖️ Comparação de Eficiência

Modo Caractere: Para digitar "hello" = 15 segmentos TCP
Modo Linha: Para digitar "hello" = 2 segmentos TCP
Diferença: 750% menos tráfego no modo linha!

📊 Análise de Overhead

Conexão via satélite (500ms ida e volta):
Modo Caractere: 500ms de atraso para cada tecla
Modo Linha: 500ms de atraso apenas no final da linha

💻 Exemplo Prático com TELNET

TELNET usa caracteres ASCII legíveis
Possível conectar diretamente via terminal
Útil para debug e testes de conectividade

🔧 Teste Manual via Terminal

$ telnet www.exemplo.com 23
Trying 198.45.24.104...
Connected to www.exemplo.com (198.45.24.104).
Escape character is '^]'.

Sistema solicita login:
Login: alice
Password: ******

Usuário executa comando:
$ ls -la
total 1024
drwxr-xr-x 5 alice users 4096 Jan 15 10:30 .
drwxr-xr-x 12 root root 4096 Jan 10 09:15 ..
-rw-r--r-- 1 alice users 156 Jan 15 10:30 arquivo.txt

$ exit
Connection closed by foreign host.

📝 Passos do Processo

1. Cliente conecta na porta 23 do servidor
2. Servidor responde com prompt de login
3. Usuário se autentica com ID e senha
4. Sistema remoto executa comandos digitados
5. Resultados são enviados de volta e exibidos
6. Conexão encerrada com exit ou logout

🔒 Questões de Segurança

TELNET transmite tudo em texto claro!
Senhas, comandos e dados são visíveis na rede
Solução moderna: SSH (Secure Shell) - porta 22
